# i18n Readiness Audit (EN→ES) — doulab.net — 2026-06-01

## Scope

Read-only audit of what it would take to ship `doulab.net` in Spanish alongside English, as a launch-eligible (not deferred) capability. Covers:

- Docusaurus i18n configuration (`docusaurus.config.ts`).
- Hardcoded English copy in `.tsx` page and component files under `src/pages/` and `src/components/`.
- MDX/MD content volume in `docs/` and `blog/`.
- Frontmatter, routing, navbar/footer labels, localized assets, date/number formatting, blog locale handling, RSS feeds, and the `write-translations` workflow.
- Reference: sister project `clients-hub/web/` which already runs EN+ES.

Out of scope: actual translation labor, glossary/style-guide construction, MT pipeline selection, CMS integration.

## Method

1. Read `AGENTS.md`, the canonical backlog (`docs/ops/doulab-net-backlog.md`), and `docusaurus.config.ts`.
2. Enumerated `src/pages/**/*.tsx` (26 files), `src/components/**/*.tsx` (13 files), `docs/**/*.md*`, and `blog/**/*.md*`.
3. Sampled high-traffic pages (`index.tsx`, `services/clarityscan.tsx`, `404.tsx`, `privacy-terms.tsx`) and shared components (`Hero.tsx`, `FinalCta.tsx`, `ConsentBanner.tsx`).
4. Word-counted MDX/MD via `wc -w`, separating concerns:
   - `src/pages/**/*.tsx`: ~27,474 words (includes JSX, props, and embedded copy; estimate ~40–55% is user-visible copy).
   - `src/components/**/*.tsx`: ~2,173 words.
   - `docs/research-resources/` (Innovation Lab Guide + MicroCanvas whitepaper, etc.): ~20,694 words.
   - `blog/` (4 posts, excluding `_backup_`): ~7,587 words.
   - `docs/intro.mdx` + `docs/releases.mdx`: ~2,811 words.
5. Compared against `clients-hub/web/docusaurus.config.ts` (i18n: `['en','es']`) and `clients-hub/web/i18n/es/` (working `navbar.json`, `footer.json`, per-plugin `current/` doc trees).
6. Inspected backlog item **B-P2-03** ("ES i18n manual plan (no Docusaurus scaffold)") — marked Completed with commit `f3f42b3a` but no `i18n/` folder exists in the repo; the plan document `docs/ops/phase-b-p2-03-notes.md` should be treated as the canonical plan, but execution has not happened.

Caveat: word counts are raw `wc` figures; translator-billable counts will be lower for `.tsx` (strips JSX/identifiers) and roughly equivalent for `.md/.mdx`.

## Findings (ranked)

### I18N-001 — Docusaurus i18n config locks site to English-only
- Severity: high. Impact: 5. Effort: S.
- Location: `doulab-site/docusaurus.config.ts:20`.
- Observation: `i18n: { defaultLocale: 'en', locales: ['en'] }`. No `localeDropdown` exists in `themeConfig.navbar.items` (lines 85–93). Build produces a single `/` tree with no `/es/...` routing path. No `i18n/` folder exists on disk.
- Recommendation: Change to `locales: ['en','es']` and add `{ type: 'localeDropdown', position: 'right' }` to navbar items. Add `localeConfigs` for `htmlLang`, `direction`, and human label (`Español`). Mirror `clients-hub/web/docusaurus.config.ts:29-32` and `:187`. This is a one-line config change that unblocks all subsequent work but is gated by the absence of translations — wire the dropdown only after the first ES page exists, to avoid empty/EN-falling-through routes.
- Evidence: `doulab-site/docusaurus.config.ts:20`, `clients-hub/web/docusaurus.config.ts:29-32`.

### I18N-002 — No `i18n/` directory exists despite backlog marking B-P2-03 completed
- Severity: high. Impact: 5. Effort: S (structural).
- Location: repo root `doulab-site/` (absent folder).
- Observation: Backlog `docs/ops/doulab-net-backlog.md:65-72` reports **B-P2-03** as Completed (`f3f42b3a`), but only a planning artifact was produced. There is no `doulab-site/i18n/es/` parallel to `clients-hub/web/i18n/es/`. The Docusaurus convention requires:
  - `i18n/es/docusaurus-theme-classic/navbar.json`
  - `i18n/es/docusaurus-theme-classic/footer.json`
  - `i18n/es/docusaurus-plugin-content-docs/current/**` (mirrors `docs/`)
  - `i18n/es/docusaurus-plugin-content-blog/**` (mirrors `blog/`)
  - `i18n/es/code.json` (UI strings emitted by `write-translations`).
- Recommendation: Run `npm run write-translations -- --locale es` to scaffold the JSON stubs, then create the docs/blog mirror manually. Backfill backlog evidence; the "Completed" status is misleading.
- Evidence: `doulab-site/docs/ops/doulab-net-backlog.md:65-72`; `clients-hub/web/i18n/es/` listing (8 plugin folders + `code.json`).

### I18N-003 — Page titles, descriptions, hero copy, and CTAs are hardcoded in TSX (largest translation surface)
- Severity: critical. Impact: 5. Effort: L.
- Location: representative — `src/pages/index.tsx:84-167` (Problem reel: 8 problem objects with cause/effect/metric/sourceLabel/pillar strings), `:243` ("Start with ClarityScan®"), `:248` ("Book a discovery call"), `:259-323` (Service pillar cards), `:331-348` (Numbers strip kpi labels), `:629-650` (Principles cards), `:656-658` (page title + description), `:705-718` (Hero props). Same pattern repeats across 25 other `.tsx` pages.
- Observation: Every visible string is a string literal in JSX, not a translation key. Docusaurus's `<Translate>` API and `translate({ id, message })` calls are not used anywhere (Grep for `localeDropdown` returned zero hits — confirming no `@docusaurus/Translate` adoption either). For ES to render, each page must either (a) be wrapped with `<Translate>` components per literal, or (b) duplicated as MDX under `src/pages/` with localized variants picked up via the Docusaurus pages plugin, or (c) the JSX content extracted into MDX under `docs/` and routed via `routeBasePath`.
- Recommendation: Adopt strategy (a) — `<Translate id="home.problem.card1.cause">…</Translate>` — for short labels (CTAs, headings, microcopy) so `write-translations` extracts them into `code.json`. For long marketing prose (Problem cards, Service descriptions, Principles), strategy (c) is better: convert each page or section to MDX islands so translators work in prose, not JSON. Plan ~3–5 days of mechanical refactor per major page.
- Evidence: `src/pages/index.tsx:84-167,243,259-323,656-718`; absence of `Translate` import in any `.tsx` file under `src/`.

### I18N-004 — Navbar and footer labels are not translation-ready
- Severity: high. Impact: 4. Effort: S.
- Location: `docusaurus.config.ts:82-135`.
- Observation: Navbar items (`'Home'`, `'What we do'`, `'Case Studies'`, `'Insights'`, `'About'`, `'Contact'`, `'GitHub'`) and footer column titles + link labels are literal English. Docusaurus auto-extracts these to `i18n/es/docusaurus-theme-classic/navbar.json` and `footer.json` via `write-translations`, but only after `locales: ['en','es']` is set (see I18N-001). The clients-hub example at `clients-hub/web/i18n/es/docusaurus-theme-classic/navbar.json` is a working reference (3 lines for a single label).
- Recommendation: After enabling `es` locale and running `write-translations`, populate the two theme-classic JSON files with translated values. Estimate ~30 strings total: ~10 minutes of translation work, plus glossary decisions ("Case Studies" → "Casos de estudio"; "What we do" → "Qué hacemos" or "Lo que hacemos" — needs Luis's call).
- Evidence: `docusaurus.config.ts:86-92,99-135`; `clients-hub/web/i18n/es/docusaurus-theme-classic/navbar.json:1-5`, `footer.json:1-20`.

### I18N-005 — Footer copyright contains live JS expression that won't localize cleanly
- Severity: medium. Impact: 2. Effort: S.
- Location: `docusaurus.config.ts:136`.
- Observation: `copyright: \`© ${new Date().getFullYear()} Doulab. MicroCanvas® and IMM® are registered marks.\`` — the year is evaluated at build time but the sentence template is a single string. Docusaurus's `write-translations` will extract this as one message key with the year already substituted, breaking year rollovers on the ES side unless the template is split.
- Recommendation: Rewrite as `\`© ${year} Doulab. MicroCanvas® and IMM® son marcas registradas.\`` for ES, or use Docusaurus `<Translate values={{year}}>` pattern. The clients-hub footer uses a frozen `'© 2025 Doulab. MicroCanvas® and IMM® are registered marks.'` (`clients-hub/web/docusaurus.config.ts:209`) which is simpler but accepts annual maintenance.
- Evidence: `docusaurus.config.ts:136`.

### I18N-006 — Blog posts have rich English-only frontmatter (tags, keywords, slugs) with no localized variants
- Severity: high. Impact: 4. Effort: M.
- Location: `blog/2026-01-19-coordination-threshold.md:1-19`, plus 3 other posts.
- Observation: Each post carries `title`, `description`, `slug`, `tags`, `keywords` (10+ items), and SEO `image` in English. ES equivalents require parallel files under `i18n/es/docusaurus-plugin-content-blog/`, preserving the same filename. `authors.yml` and `tags.yml` also need ES variants — Docusaurus expects them at `i18n/es/docusaurus-plugin-content-blog/options.json` (categories/sidebar labels) plus per-locale `authors.yml`/`tags.yml`.
- Recommendation: Translate slugs to ES (e.g., `coordination-threshold-ecosystems` → `umbral-de-coordinacion-ecosistemas`) for SEO; keep filenames identical for Docusaurus mapping. Decide whether tags translate or stay canonical English (recommend: stay English to preserve cross-locale tag pages, but localize `tagline`/labels). Word count: ~7,587 English words across 4 posts, so ~7,500–8,500 ES words.
- Evidence: `blog/2026-01-19-coordination-threshold.md:1-19`.

### I18N-007 — Date formatting is hardcoded to `en-US` locale
- Severity: medium. Impact: 3. Effort: S.
- Location: `src/pages/insights/index.tsx:146`.
- Observation: `date.toLocaleDateString('en-US', { year: 'numeric', month: 'short', day: '2-digit', timeZone: 'Europe/Zurich' })` — Spanish visitors will see "Jan 19, 2026" instead of "19 ene 2026". Single occurrence found, but pattern is the only date format in the codebase, making the fix narrow.
- Recommendation: Use the Docusaurus `useLocation`/`useDocusaurusContext` hook to read `i18n.currentLocale` and pass it into `toLocaleDateString(locale, …)`. Apply same fix anywhere date strings appear in components (none found beyond this one).
- Evidence: `src/pages/insights/index.tsx:144-146`.

### I18N-008 — RSS/Atom feeds and `useLatestBlogPosts` fetch hardcoded `/blog/rss.xml`
- Severity: medium. Impact: 3. Effort: S.
- Location: `src/pages/index.tsx:372`; `docusaurus.config.ts:44` (`feedOptions: { type: 'all' }`).
- Observation: With i18n enabled, Docusaurus generates per-locale feeds at `/blog/rss.xml` (en) and `/es/blog/rss.xml`. The homepage RSS-loader code at `src/pages/index.tsx:372` fetches the literal `/blog/rss.xml` regardless of current locale; on `/es/`, this will show English posts in the "Latest from the blog" card.
- Recommendation: Read `useDocusaurusContext().i18n.currentLocale` and prefix the feed URL accordingly (`/es/blog/rss.xml` when not default). Also expose both feeds in the footer (currently only the EN feed is linked, `docusaurus.config.ts:130-131`).
- Evidence: `src/pages/index.tsx:362-423`; `docusaurus.config.ts:42-50,130-131`.

### I18N-009 — Image assets and OG cards are English-text-baked; no ES variants exist
- Severity: medium. Impact: 3. Effort: M.
- Location: `static/img/social/` (`default-og.jpg`, `og-imm-program.jpg`, `og-insights.jpg`); blog hero images under `static/img/blog/2026/`; case-study card images.
- Observation: Open Graph cards embed English titles directly into the JPG/AVIF/WebP triples. Frontmatter (`image:` keys in blog and `og:image` `<meta>` in TSX pages, e.g., `index.tsx:667`) doesn't switch by locale today.

### I18N-010 — Long-form research docs (Innovation Lab Guide, MicroCanvas v2.1) dominate translation cost

Keep `docs/ops/**` English-only (internal ops, never user-facing).

### I18N-011 — `docs/ops/**` and other internal docs would be incorrectly indexed for translation
- Severity: medium. Impact: 3. Effort: S.
- Location: `docs/ops/` (~55 ops files including this audit folder).
- Observation: When `i18n` is enabled, `docusaurus-plugin-content-docs` will mirror the entire `docs/` tree under `i18n/es/docusaurus-plugin-content-docs/current/`, including operational notes that have no user value in any language. Without filtering, translators will see ~100k extra words of ops noise.
- Recommendation: Use the docs plugin `exclude` option to drop `ops/**` from the public docs build, or move ops files outside `docs/` (e.g., `ops/` at repo root). Either approach also simplifies the docs sidebar.
- Evidence: `docusaurus.config.ts:34-41` (no `exclude` set); `docs/ops/` listing in `AGENTS.md:54-104`.

### I18N-012 — Embedded brand-mark sentences will need ES copy decisions (registered marks, IMM-P®, MicroCanvas®)
- Severity: low. Impact: 2. Effort: S.
- Location: `docusaurus.config.ts:136`; `src/pages/index.tsx:243,294,489`; multiple service pages.
- Observation: Phrases like "Start with ClarityScan®", "MicroCanvas® and IMM® are registered marks", "Built on MicroCanvas® v2.1 and IMM-P® gates" embed registered marks into running prose. ES translation must preserve the marks exactly (Unicode `®` codepoint, no spacing changes) and decide on standard ES phrasings ("Empieza con ClarityScan®", "MicroCanvas® e IMM® son marcas registradas" — see how clients-hub renders it: `clients-hub/web/i18n/es/docusaurus-theme-classic/footer.json:18`).
- Recommendation: Lock a one-page ES brand glossary before translation starts. Reuse the clients-hub phrasing as the source of truth.
- Evidence: `docusaurus.config.ts:136`; `clients-hub/web/i18n/es/docusaurus-theme-classic/footer.json:17-19`.

### I18N-013 — `<Layout>` `title` and `description` props bypass translation extraction
- Severity: medium. Impact: 3. Effort: M.
- Location: `src/pages/index.tsx:661`; `src/pages/services/clarityscan.tsx:39-41`; pattern repeats in every page.
- Observation: `<Layout title="..." description="...">` accepts plain strings that drive `<title>` and `<meta description>`. Docusaurus does not auto-extract these via `write-translations` because they are runtime props, not `<Translate>` calls. Same applies to all `<meta property="og:*">` content in `<Head>` blocks (e.g., `index.tsx:665-668`).
- Recommendation: Replace with `translate({ id, message })` calls so they end up in `code.json`. Roughly 26 pages × 4 strings (title, description, og:title, og:description) = ~104 keys.
- Evidence: `src/pages/index.tsx:656-669`; `src/pages/services/clarityscan.tsx:39-48`.

### I18N-014 — JSON-LD schema fields (descriptions, names) are hardcoded English
- Severity: low. Impact: 2. Effort: S.
- Location: `src/pages/services/clarityscan.tsx:25-36`; `src/pages/index.tsx:670-690`.
- Observation: Schema.org `Service.description`, `Organization.name`, search-action targets contain English-only literals. For SEO on the ES tree these should be localized so Google indexes Spanish rich-snippet descriptions; technical schema keys (`@type`, `@context`) stay English.
- Recommendation: Build the schema object inside the component using `translate()` calls for `name`, `description`, and `serviceType`. Add `inLanguage: 'es'` per locale.
- Evidence: `src/pages/services/clarityscan.tsx:25-36`.

### I18N-015 — No `hreflang` alternate links emitted in `<Head>` for cross-locale SEO
- Severity: medium. Impact: 3. Effort: S.
- Location: every page's `<Head>` (e.g., `src/pages/index.tsx:662-700`).
- Observation: Once both locales exist, Google needs `<link rel="alternate" hreflang="en" href="https://doulab.net/..." />` and `hreflang="es"` pairs on every page to avoid duplicate-content penalties and to surface the right locale per region. Docusaurus v3 auto-adds these when i18n is configured, but only for routes that exist in both locales — partial coverage will degrade signals.
- Recommendation: Verify `<DocusaurusContext>`-provided alternates appear in built HTML after enabling i18n. Set the `x-default` hreflang to EN. Cross-check with the SEO hand-off (file `13-seo.md`).
- Evidence: `src/pages/index.tsx:662-700` (no hreflang block).

## Quick wins — top 5

1. **Flip the config and scaffold** (I18N-001, I18N-002): set `locales: ['en','es']`, add `localeDropdown`, run `npm run write-translations -- --locale es`. Half a day. Unblocks everything downstream. Do NOT deploy until at least navbar/footer/home are translated, or the locale dropdown will dead-end.
2. **Translate navbar + footer JSON** (I18N-004): ~30 strings, ~10 min translation + ~10 min review. Immediate visible bilingual surface.
3. **Exclude `docs/ops/**` from the docs plugin** (I18N-011): add `exclude: ['ops/**']` to `docusaurus.config.ts:36`. One-line change; saves ~100k words of irrelevant translator-facing surface.
4. **Fix the `en-US` date hardcode** (I18N-007): single-file edit at `src/pages/insights/index.tsx:146` to use `currentLocale`.
5. **Lock the ES brand glossary** (I18N-012): one-page decision doc reusing `clients-hub` phrasings. Prevents glossary drift across translators and prevents ®-mark corruption.

## Strategic bets — top 3

1. **Adopt `<Translate>` + MDX-island hybrid for `.tsx` pages** (I18N-003, I18N-013). Wrap short labels (CTAs, headings, microcopy) with `<Translate>` so they extract to `code.json`; refactor long prose sections (Problem cards, Service pillars, Principles, Numbers strip notes) into MDX components imported by the page, so translators work in Markdown not JSON. This is the only sustainable answer for a 27k-word TSX surface. Budget: 2–3 engineer-weeks for refactor + ~6,000–8,000 word ES translation pass for launch scope.
2. **Phased rollout by route, not by component** (I18N-010). Ship ES for `/`, `/what-we-do`, `/services/*`, `/case-studies/*`, `/contact`, `/privacy-terms`, plus the navbar/footer — call this launch. Phase 2: blog backlog (~7,600 words). Phase 3: Innovation Lab Guide + whitepapers (~20,700 words). Phase 4: research-resources index. This avoids the "everything or nothing" trap and lets the locale dropdown go live with full coverage of the launch surface only.
3. **Localized social/OG assets and per-locale RSS** (I18N-008, I18N-009, I18N-015). The site's organic-search and LinkedIn surfaces are LatAm/Caribbean-facing; English social cards on `/es/` routes will undermine credibility. Bundle: ES OG cards (~10–15 images), per-locale RSS feeds wired into the homepage card, hreflang verification. Hand-off to Brand & visual + SEO.

## Out of scope / hand-offs

- **IA & UX**: locale-switcher placement, ES-aware breadcrumbs, ES-aware page metadata defaults.
- **Brand & visual**: ES OG cards, ES blog hero banners (NNY hero system, see `docs/ops/design-nny-hero-language.md`), ES typography pairings if any glyph coverage gaps emerge.
- **Content & copy**: actual EN→ES translation, glossary, registered-mark phrasing standardization, voice/tone calibration for LatAm/Caribbean audiences.
- **Conversion**: ES CTA copy A/B (e.g., "Reserva una llamada" vs "Agenda una llamada de descubrimiento"); ES booking flow audit on `booking.doulab.net` (the booking domain is separate but its copy must match).
- **SEO**: hreflang validation, ES sitemap entries, ES keyword research, ES schema.org `inLanguage` audit (cross-ref I18N-015).
- **Performance**: ES build adds duplicate routes and asset URLs — verify Cloudflare cache strategy and build-time impact.
- **Accessibility**: ES `aria-label`s and `lang` attributes on mixed-language fragments; screen-reader voice changes.
- **Security & privacy**: ES privacy-terms.tsx — `legal@doulab.net` contact stays, but jurisdictional language ("Santiago Arias Consulting") may need ES-locale clarifications.
- **Analytics**: locale dimension in event tracking (`data-cta` values currently mix EN literals; consider splitting by locale).
- **Code quality**: enforce `<Translate>`/`translate()` adoption via lint rule that flags string literals in JSX text content under `src/pages/` and `src/components/`.
- **Blog & editorial**: ES editorial calendar, ES author bios in `blog/authors.yml`, ES tag taxonomy decision.
- **MCF/IMM-P**: ES reference card naming, alignment with existing ES decks/videos.
- **Behavioral economics / Behavioral psychology**: ES microcopy choices ("¿Listo para…" vs imperative "Empieza con…") — material conversion effect.
- **Sales**: ES booking confirmations, ES email templates triggered by `booking.doulab.net`.
- **Mobile-first**: ES strings tend to be 15–25% longer than EN — verify button/CTA wrapping on small viewports for the longest strings ("Reservar una llamada de descubrimiento", "Empieza con ClarityScan®").
